<decisions>
## Implementation Decisions

### Pin strategy — Drop per-node suffix (primary) with node-pinning fallback

- **Primary approach**: Remove the `${SLURMD_NODENAME:-local}` suffix from `JAX_COMPILATION_CACHE_DIR`. Cache dir becomes a single shared path (e.g., `/scratch/${_PROJECT}/${USER}/.jax_cache_gpu/shared/`). Scheduler remains free to place jobs on any available GPU node; cache reuse happens across nodes via the shared Lustre mount.
- **Why user chose this**: "I'd prefer to drop per-node, that way we can optimally schedule across nodes and reuse the cache IF possible." — preserves scheduler flexibility, doesn't lock a single node.
- **Known prior attempt**: User recalls having tried a single-subprocess restructure before and hitting "similar problems" — do NOT re-attempt single-subprocess as the primary mechanism.
- **Fallback**: If the verification P-sweep shows either (a) no speedup (concurrent-write contention or Lustre lock thrash) or (b) corrupt cache entries, fall back to `--nodelist=<pinned>` option and document the regression. Capture the fallback in VERIFICATION.md — don't silently revert.
- **Researcher should investigate**: JAX persistent-cache file-lock behaviour on Lustre; whether JAX has a documented concurrent-write-safety mode; existing blackjax/pyhgf SLURM examples that use shared network mounts for the cache dir.

### N=2 warm-up block — Replace with N=50 dry-run in same process

- **Delete the N=2 compile step.** Phase 21 §21-06 evidence: N=2 scan-body sha256 (`2f25aed94588c656`) ≠ N=50 sha256 (`62fbc95475cb0a32`), Jaccard 0.180 — kernels are structurally orthogonal.
- **Replace with N=50 dry-run**: within the SAME Python subprocess as the real measurement, run one N=50 forward pass (JIT compile only, no sampling), discard result, then run the real N=50 benchmark. The in-process JAX tracing cache (proven 2–2.5× speedup at P=5/50 in Phase 21) handles the warm measurement regardless of whether the persistent on-disk cache is working.
- **Pairs with primary pin strategy**: N=50 dry-run is an in-process belt-and-suspenders for the case where the persistent shared-cache approach doesn't land cleanly; correctness does not depend on cross-process cache reuse.
- **Implementation sketch**: add a `--dry-run-warmup` flag (or reuse existing `--dry-run` if semantics align) to `scripts/08_run_power_iteration.py` that runs the compile path at the given P and exits. SLURM calls it once before the real `--benchmark` invocation in the same bash job. Cheaper than a second SLURM submission; keeps walltime budget tight.

### Verification plan — Both diagnostic reverify AND production direct submission

- **Step 1 (diagnostic gate)**: `bash cluster/21_submit_pscan_chain.sh` with the patched SLURM infrastructure. Gate criterion: at P=50, `next_proc_warm_s ≤ 75s` (target matching the proven same_proc_warm=93s ± tolerance), which is `next_proc_warm/cold ≥ 3×`. If gate fails, trigger the `--nodelist` fallback and re-run.
- **Step 2 (production confirmation)**: `sbatch cluster/14_benchmark_gpu.slurm` with production partition/walltime. Gate criterion: job completes under walltime AND writes `results/power/benchmark_batched.json`. This is the actual 14.1-03 deliverable — 14.1-07 covers the prerequisite, 14.1-03 owns the benchmark write itself.
- **Sequencing**: Step 1 must pass before Step 2 submission (cheap cluster time before expensive cluster time). Plan may split into two commits: patch + Step 1 pass → 14.1-07 SUMMARY; Step 2 success → 14.1-03 SUMMARY.
- **Both gates documented in 14.1-07 VERIFICATION.md** with quantitative evidence (jit_scaling_sweep.csv updated row + benchmark_batched.json shape/metadata).

### Sibling SLURM scope — Apply fix repo-wide

- **All SLURM scripts using `${SLURMD_NODENAME}`** get the cache-dir-suffix drop. User directive: "we should make this fix throughout the repo" — prefer consistent cluster infrastructure over per-script idiosyncrasy.
- **Scripts touched** (researcher/planner should `grep -l SLURMD_NODENAME cluster/*.slurm` as the authoritative list; expected targets include):

<specifics>
## Specific Ideas

- **Verification numeric target for Step 1 gate**: at P=50, `next_proc_warm_s / cold_jit_s ≥ 3.0×` (baseline today is 1.00× per `jit_scaling_sweep.csv`). Concretely: `next_proc_warm_s ≤ 75s` (vs `cold_jit_s ≈ 229s`), matching the proven in-process `same_proc_warm_s ≈ 93s`.
- **Regression guard**: P=5 same_proc_warm speedup must remain ≥ 2.0× (no regression from the repo-wide cache-dir change).
- **Fallback trigger**: if Step 1 shows either `next_proc_warm/cold < 2.0×` OR corrupt cache-hit log entries (e.g., "cache entry version mismatch"), switch to `--nodelist=<pinned>` on the benchmark script only; keep shared-cache on other SLURM scripts.
- **User stated constraint**: single-subprocess restructure was attempted previously and had "similar problems". The N=50-dry-run-in-same-subprocess pattern this plan uses is different — it does NOT eliminate the persistent cache dependency (shared cache is still the primary cross-job mechanism); it's an in-process belt that makes correctness robust to persistent-cache failure modes. Researcher should surface any old SUMMARY/STATE entries documenting the prior single-subprocess attempt for review.
- **Cluster file system is Lustre** (M3 MASSIVE); Lustre's `flock` semantics are slower than local ext4 but JAX's file-locking is coarse-grained (once per compile, not once per op). Contention likelihood is low for the 14.1-03 benchmark scale but should be confirmed in verification.

</specifics>
