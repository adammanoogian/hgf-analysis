# Phase 14.1-07 — Node-Pin Gap Closure (SUMMARY)

## Outcome

Phase 21 VERDICT §(c) identified the dominant benchmark bottleneck as per-node JAX
compilation-cache scope: `JAX_COMPILATION_CACHE_DIR` was routed through
`/.jax_cache_{gpu|cpu}/${SLURMD_NODENAME:-local}`, so each SLURM job landing on a
different node found an empty cache regardless of prior warm-up runs. Plan 14.1-07a
applied a repo-wide infrastructure patch (atomic commit `6b7e6ed`), dropping the
`${SLURMD_NODENAME:-local}` suffix in all nine GPU/CPU SLURM scripts and upgrading
the Phase-1 warm-up cohort from N=2/group to `N=max(n_per_group_grid)` so the
in-process tracing cache matches the Phase-2 production shape. Plan 14.1-07b submitted
a diagnostic P-sweep chain (6 SLURM jobs, P ∈ {5, 20, 50}) and confirmed cross-process
+ cross-node XLA persistent-cache reuse at the XLA layer. The active strategy is
`shared-dir`; Task 3 (nodelist-fallback) was skipped on the strength of the 10/10
cross-node cache-hit evidence. Phase 14.1-03 is now unblocked.

## Gate Verdict Note

The literal VERDICT §e Primary gate (≥ 3.0× at P=50) fails at **1.04×** — not because
the fix failed, but because the gate criterion was miscalibrated: XLA compile is only
~1.5% of P=50 walltime, so even perfect compile caching cannot produce a 3.0× walltime
speedup. The authoritative evidence and the full derivation of why this is a gate
miscalibration rather than a mechanism failure are in:

```
.planning/phases/14-integration-gpu-benchmark/14.1-07-VERIFICATION.md §Gate Criterion Caveat
```

The summary: persistent cache is now hitting 10/10 cross-process AND cross-node (m3g111
cold → m3g108 warm); the remaining ~220s per cold process boot is JAX tracing + Python
imports + blackjax window adaptation, which persistent-compile cache cannot address by
definition. The fix is real; the gate was wrong.

## Active Infrastructure Strategy

**`shared-dir`** — all 9 GPU/CPU SLURM scripts now route `JAX_COMPILATION_CACHE_DIR`
to `/scratch/${_PROJECT}/${USER}/.jax_cache_{gpu|cpu}[/subdir]/shared` on the Lustre
scratch mount. The SLURM scheduler remains free to assign any available node; cross-node
cache reuse is provided by the shared Lustre path rather than node pinning. No
`--nodelist` directive was added to any SLURM script. The ArviZ import-time race
(concurrent `sbatch` writes to `$HOME/.cache/arviz/daily_warning`) is defeated by
writing today's ISO date into the stamp file in every patched SLURM header before
Python runs.

Shared-dir was chosen over nodelist-fallback because the mechanism evidence from
14.1-07b (10/10 cross-node hits at P=50) proved that cache-key scope was the only
broken invariant; the cache keys themselves are node-agnostic. Node pinning would
address a different, non-existent problem. See VERIFICATION.md for the Lustre
contention probe confirming no flock/cache-corruption symptoms at this workload scale.

## Deferred Options

**AOT compile + serialize** (`jax.jit(...).lower().compile()` + `jax.export.serialize()`) is the next lever to pull if 14.1-03's walltime materially exceeds the 50 GPU-hour/chunk threshold despite this fix.

Rationale (from VERIFICATION.md §Gate Criterion Caveat): the ~220s cold-boot walltime at P=50 splits into ~5–10s Python imports + ~100s JAX tracing + ~3s XLA compile + ~100–120s BlackJAX window adaptation. Persistent compile cache is now saturated (XLA compile = ~1.5% of walltime). The only addressable layer for further cross-process savings is the ~100s JAX tracing, which AOT serialization would bypass entirely.

At chunk-based 14.1-03 scale (each chunk is one Python process, ~1400 iterations), tracing re-runs once per chunk start — so AOT would save ~5 min total across a 3-chunk benchmark, which is unlikely to move the GPU-hours decision. AOT is therefore explicitly deferred until 14.1-03 data reveals an actual problem. Decision rule: if `total_chunk_walltime` puts the benchmark over the 50 GPU-hour threshold AND the excess is attributable to per-chunk tracing overhead (not sampler cost), then AOT is the targeted fix. This requires a research spike on shape-variation + blackjax internals + M3 JAX version compatibility before implementation.
