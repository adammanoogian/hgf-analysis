## Discovered During Execution (pre-existing issues fixed in-plan)

**ArviZ import-time race.** The first chain submission (jobs 54902365..54902370, pre-arviz-fix) lost 4/6 jobs to `FileNotFoundError` at `arviz/__init__.py:36 _atomic_write_text`: concurrent `sbatch` launches raced `tmp.replace(path)` on the shared `$HOME` mount. Defeated by writing today's ISO date into `$HOME/.cache/arviz/daily_warning` in every patched SLURM header before any Python runs (`_warn_once_per_day` takes the early-return branch when the stamp matches today). Documented in Obsidian vault: `_inbox/JAX Persistent Compilation Cache - Lessons from SLURM GPU Deployment - 2026-04-20.md`.

## AOT Compile + Serialize — DEFERRED (not in 14.1-07 scope)

The ~220s residual cold-boot walltime at P=50 is tracing + imports + blackjax window adaptation sampling. Persistent compile cache cannot address any of these. The sharpest available lever for further walltime reduction is AOT compile via `jax.jit(...).lower().compile()` + `jax.export.serialize()`, which would bypass the ~100s tracing layer on each new process. Explicitly deferred because:

1. Not in VERDICT §(c) — would expand 14.1-07's charter.
2. Uncertain value at 14.1-03's chunk structure (each chunk is one long Python process running ~1400 iterations; tracing re-runs only per chunk start, so AOT saves ~5 min total across a 3-chunk benchmark).
3. Requires a research spike on shape-variation + blackjax internals + M3's installed JAX version.

**Decision rule:** re-evaluate only if 14.1-03 materially exceeds the Phase-14 decision-gate criterion (STATE #210: > 50 GPU-hours/chunk → fall back to CPU `comp` partition). See `14.1-07-VERIFICATION.md` §Gate Criterion Caveat and §Flag for monitoring.
