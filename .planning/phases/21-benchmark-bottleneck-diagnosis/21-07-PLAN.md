  Step 2 — Adjudicate candidates:
  For each of the 7 bottleneck candidates, write a paragraph evaluating evidence:

    1. NUTS sampler step function (leapfrog / warmup):
       Evidence for: NUTS cold_s / Laplace total_s ratio at matched P. If >3x,
       NUTS adds meaningful cost.
       Evidence against: If Laplace is ALSO slow (total_s > 60s at P=5), the
       scan body dominates regardless of sampler.
       Verdict bias: if NUTS/Laplace >> 1, support sampler; if NUTS/Laplace ~1,
       refute sampler.

    2. HGF scan graph size:
       Evidence for: 21-01 scope memo scan-body depth (~100-140 ops/trial vs
       NumPyro HMM ~200 but with no nested @jit — so pyhgf's compilation cost
       may be higher per trial due to nesting). HLO op count at P=50
       extrapolated >> state-space baselines.
       Evidence against: if same-proc in-process warm speedup is large (e.g.
       cold_s/same_proc_warm_s > 10x at P=5), the tracing cache works, meaning
       graph size itself is tractable once traced.
       Verdict bias: supported only if both (a) HLO op count is large AND (b)
       same-proc warm speedup is modest (<5x).

    3. JIT cache key instability (closure-over-data):
       Evidence for: next_proc_warm_s / cold_jit_s ratio close to 1.0 at ALL P
       (persistent cache never hits). HLO hash matrix from 21-06 shows
       pscan_P5_cold vs pscan_P5_warm (different SLURM processes) have
       DIFFERENT hashes. Matches RESEARCH Risk #5 pattern.
       Evidence against: HLO hashes identical across SLURM processes at the same
       P (pscan_P5_cold == pscan_P5_warm hash-wise).
       Verdict bias: this is the lead hypothesis per RESEARCH. Confirm with
       hash-matrix evidence.

    4. Cache initialization bug:
       Evidence for: cache-dir listing shows 0 or very few files after Wave-2
       jobs completed. Matches RESEARCH §Known Pitfalls #6.
       Evidence against: listing shows populated kernel binaries.
       Verdict bias: low prior (SLURM env var is set before Python starts per
       14_benchmark_gpu.slurm lines 17-50). Only supported if cache dir is
       genuinely empty after Wave 2.

    5. Tracing-cost-dominates paradox (JAX #22281):
       Evidence for: HLO hashes ARE stable across processes (= cache hits should
       fire) AND next_proc_warm_s is still slow (> 30% of cold_jit_s). Matches
       RESEARCH §"Cache Layers (Two-Layer System)".
       Evidence against: next_proc_warm_s << cold_jit_s (cache is saving time).
       Verdict bias: this is the fallback hypothesis if candidate #3 is refuted.

    6. Shape polymorphism failure (Phase-1 N=2 not warming N=50):
       Evidence for: pscan_P5_cold vs pscan_P50_cold HLO hashes are different
       (expected by vmap) AND Phase-1 of 14_benchmark_gpu.slurm uses N=2 while
       Phase-2 uses N=50.
       Evidence against: pscan_P5_cold == pscan_P50_cold (would mean vmap is
       shape-polymorphic, which would be positive news but is unexpected).
       Verdict bias: this IS the Phase-1-vs-Phase-2 structural issue. This is
       NOT a "bug" per se — it's the bench design being wrong. This bottleneck
       has a clean fix: rewrite Phase 1 to run at P=50 directly.

    7. Other / combination:
       - Warmup-params duplication (RESEARCH Risk #4): 21-02 either fixed this or
         surfaced it as a blocker. If 21-02 SUMMARY said fit_batch_hierarchical
         does NOT accept warmup_params, the 1.1x speedup is partially explained by
         NUTS window adaptation running twice (adding ~1100s to the warm call).
         This confounds the 1.1x observation and must be flagged in the verdict.
       - "both": if Laplace AND NUTS show the same HLO-hash instability pattern,
         the fix is cache-key-level; if Laplace is fast and NUTS is slow, the
         sampler needs attention too.
