Run the VB-Laplace JIT probe on M3 GPU at P ∈ {5, 20} and produce the `vb_laplace_vs_nuts_jit.json`

forensics) and 21-07 (VERDICT) synthesize.

**In scope:**
- New SLURM script `cluster/21_diagnose_laplace.slurm`:
    - Walltime `#SBATCH --time=00:30:00` (SC7 compliant).
    - Same GPU allocation, conda env, PATH setup, JAX_COMPILATION_CACHE_DIR as
      `cluster/14_benchmark_gpu.slurm` for apples-to-apples reproducibility.
    - Sets `JAX_LOG_COMPILES=1` at SLURM env level.
    - Sets `XLA_FLAGS=--xla_dump_hlo_as_text --xla_dump_to=<archive_dir>` where
      `<archive_dir>` is unique per-P (so P=5 and P=20 dumps don't overwrite each
      other).
    - Invokes `python scripts/08_run_power_iteration.py --vb-laplace-probe --probe-p 5
      --diagnostic-mode` then `--probe-p 20 --diagnostic-mode` sequentially within
      one job.
    - After the probes finish, copies the XLA text-IR dumps into
      `.planning/phases/21-benchmark-bottleneck-diagnosis/hlo_dumps/laplace_P{5,20}/`
      (text-IR archived per-run; CONTEXT.md decided on "text IR archived per-run;
      heavy dumps replace latest only" — text-IR is cheap so archiving both P values
      is fine).
    - Ends with a git auto-push block (mirror 14.1-01 pattern in
      cluster/14_benchmark_gpu.slurm) that commits the new artifacts + SLURM
      stdout/stderr logs + updated JSON and pushes to origin/main.
- A `checkpoint:human-action` task for submission. Per the planner spec, cluster
  submissions are never autonomous; the user sbatch's the job, waits, then resumes
  with the job ID + outcome.
- A follow-up auto task that (after the user resumes) verifies the artifacts landed on
  local after `git pull origin main` and records headline numbers into
  `21-03-SUMMARY.md`.

**Explicitly NOT in scope:**
- Running the NUTS-matched probe — plan 21-04 owns that. The two probes are split to
  preserve per-stage timing fidelity (NUTS requires window-adaptation handling;
  Laplace does not).
- Running P=50 in this plan. CONTEXT.md "early-stop at verdict" policy: P ∈ {5, 20} is
  sufficient to establish the scaling curve for the Laplace path. If 21-06/21-07
  require P=50 Laplace evidence to adjudicate the verdict, that's a follow-up plan.
- Running P=2. The scaling from 5 → 20 is more informative than 2 → 5 because
  P=2 is already tiny and any cold compile time at P=2 is dominated by constants.
- Implementing the closure-vs-traced-args fix (VERDICT-named, out of Phase 21 scope).
- Writing/modifying `results/power/benchmark_batched.json` (14.1-03 owns it).
- Changing any file in `src/prl_hgf/fitting/` (diagnostic-only scope).

Purpose: The VB-Laplace probe is the **logp-graph-vs-sampler isolation test** per
CONTEXT decision. If LBFGS + Hessian compile slowly, the logp graph is the
bottleneck (verdict candidate: rewrite scan body or port closure→traced-args). If
LBFGS + Hessian compile fast, the logp graph is NOT the bottleneck and the NUTS
probe (plan 21-04) will name the sampler as the culprit. Either way, 21-03 produces
one half of the data VERDICT.md synthesizes.

Output:
- `cluster/21_diagnose_laplace.slurm` committed and pushed to origin/main (so the
  cluster can `git pull` it).

- Text-IR HLO dumps under `hlo_dumps/laplace_P{5,20}/`.
- `.planning/phases/21-benchmark-bottleneck-diagnosis/21-03-SUMMARY.md` with headline
  numbers (cold total_s, compile-event counts, cache hit/miss counts, Laplace-path-level
  observation on whether LBFGS or Hessian dominates compile time).
